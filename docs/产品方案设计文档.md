> 第五届研究生金融科技创新大赛 · 东吴证券 | 2026-07-19 | 基于 V12 综合设计方案
>
> 面向个人投资者的财报反欺诈智能问答助手。本文档按照产品设计文档模板编排，覆盖产品概述、功能模块、前后端设计、API 接口、交互时序和技术选型，供语雀发布及团队开发参考。

## 1. 产品概述

### 1.1 产品定位

**产品名称**：织网鉴真 · 财报反欺诈智能问答系统

**产品定位**：面向个人投资者的财报反欺诈智能问答助手。用户通过自然语言询问上市公司的财务异常、股权关系、关联方风险、公告舆情和行业位置，系统以确定性计算与 Agentic AI 协同方式完成分析，返回带数据口径、时间范围、证据链、风险等级和不确定性说明的结果。

**核心表达原则**：

- 系统识别的是"异常信号"和"风险模式"，不是代替监管、审计或司法机构作出造假认定；
- 数值结论来自规则引擎、图算法和统计程序；
- LLM 负责意图理解、信息组织和自然语言解释；
- 每项事实性结论必须能够回到结构化数据或原始文本证据。

### 1.2 目标用户

| 用户类型 | 主要需求 | 产品响应 |
|------|------|------|
| 个人投资者 | 快速判断公司是否存在值得关注的风险 | 自然语言诊断、风险摘要和证据链 |
| 财经爱好者 | 理解财报指标和异常逻辑 | 规则解释、趋势图、行业分位 |
| 金融专业学生 | 学习股权穿透和财务勾稽 | 可视化路径、公式来源和交叉验证 |
| 竞赛评审人员 | 验证系统完整性、创新性和可解释性 | 完整演示闭环、可复现结果和性能指标 |
| 项目研究人员 | 进行公司风险初筛和案例研究 | 企业画像、跨公司对比和报告导出 |

### 1.3 赛题背景

面向 **个人投资者** 的财报反欺诈智能问答助手。用户用自然语言提问，系统穿透股权关系、交叉验证财务数据、对齐舆情事件，给出带证据链的分析回答。

| 痛点                 | 赛题原文                       | 我们的映射                |
| -------------------- | ------------------------------ | ------------------------- |
| 长对话"记忆遗忘"     | 超10轮忘上下文，不知何时调工具 | 编排 Agent + 记忆 Agent   |
| 隐性资本关联"穿透难" | 间接控股、交叉持股到不了底层   | 股权穿透 Skill（Neo4j）   |
| 财报粉饰"隐蔽性高"   | 单表完美，需跨科目勾稽         | 财务勾稽 Agent（7条规则） |

> 补充痛点：舆情碎片化（舆情事件 Skill）、分析无证据（问答生成 Agent + 证据链）

### 1.5 评分方式

赛方回信确认（2026-07-07）：无标准答案，语义+结果双维评分。

```
总分 = 0.5 × 多轮QA平均准确率 + 0.5 × 最终复杂Query准确率
单项准确率 = 0.6 × 结果正确 + 0.4 × 语义相似
```

> 语义相似度占40%，最终复杂Query（think_flag=True的77道深度题）占50%权重。

### 1.7 核心创新

前三条为赛题框架外的主动构建，后三条来自竞品调研（6款产品）的功能缝合。

**1. 多维度交叉验证。** 任一模块发现异常信号时，编排 Agent 自动触发其余两个模块进行链式核查——财务勾稽发现应收暴增 → 查股权中大股东是否在减持 + 查舆情中是否有相关负面公告 → 三线汇聚输出综合置信度。三个维度信号单看都是"可能有问题"，撞在一起形成相互印证。具体场景：其他应收款畸高→查股权链→实控人同时控制C/D公司→C/D也有财务异常→资金占用嫌疑链。

**2. 造假模式识别：类型 + 阶段。** 多条规则同时触发时，将异常信号组合反向匹配到已知造假手法（如"应收暴增+现金流为负+毛利率异常高"→"疑似虚增收入型造假"）。同时利用10个季度数据检测造假时序：应收/营收先背离 → 现金流滞后恶化 → 大股东开始减持/质押 → 监管介入。不仅回答"有风险"，还回答"现在处于早期还是高危阶段"。

**3. 行业智能对标：均值 + 历史参照。** 将通用阈值对照申万行业内百分位，输出三层信息：当期数值+历史趋势折线+行业分位。更进一步引入"同类暴雷参照"——当一家公司异常信号组合与历史上某家造假公司相似时自动提示参照。

**4. 多源风险评分体系。** 引入三层风险标签（负面诚信/监管处罚/诉讼仲裁→子类→具体事件），MVP选15-20个核心标签。综合风险由三路信号五等级评定（红橙黄蓝绿）：公告情绪映射、研报评级拐点（同一季度≥2家下调→橙色）、财务勾稽规则触发数（≥3条→红色）。

**5. 企业画像页 + 图谱风险可视化。** 围绕目标上市公司建立统一入口，把股权穿透、财务异常、舆情事件、关联风险统一展示。图谱节点按类型挂标签，异常节点按规则触发数标红/黄，形成"风险热力图"。

**6. 结构化问答与递进交互。** 回答输出排查表+证据链（按财务勾稽/关联风险/舆情事件分类折叠，每项挂来源引用）。交互上参考问财6轮递进模式，引导用户从"有风险吗"逐步深入到"哪里有问题→和同类比呢→又该如何"。

### 1.8 产品负责与不负责

**系统负责**：

- 基于当前数据识别异常信号；
- 给出规则、图谱、事件和行业对标证据；
- 展示时间范围、数据版本和缺失情况；
- 支持多轮追问、主体消歧和报告生成；
- 在模块超时或数据不足时返回部分结果；
- 区分确定事实、算法判断、LLM 解释和建议性追问。

**系统不负责**：

- 提供具体买卖指令；
- 将风险信号直接认定为财务造假；
- 在无数据来源时生成具体数值或关系；
- 用 LLM 替代财务计算、持股连乘和行业统计；
- 把事件时间先后默认标记为因果；
- 对不在数据覆盖范围内的公司伪造完整画像。

| 企业画像页底部 | 点击"对比同行 →" | 跨公司对比页 | **打开新页面**，自动带入当前公司 |
| 企业画像页 | 点击"导出报告" | 报告任务页 | **打开新页面** |

**设计原则**：
- 对话主页是唯一入口，不设全局导航栏——用户始终从对话出发；
- 画像/对比/报告均为独立全屏页，不是弹窗或面板子页面；
- 所有跳转均为"打开新页面"，各页面顶部设返回按钮；
- 返回时上一页状态保留（对话历史、滚动位置、已加载数据不丢失）。

参考企查查/天眼查的统一入口设计，围绕目标上市公司建立一张信息聚合页，将基本信息、风险标签、财务异常、股权穿透、舆情时间线、证据引用集中展示。详细页面设计见 [§4.6 企业画像页](#46-企业画像页)。


### 2.4 交互逻辑

| 场景               | 用户操作                                   | 系统响应                                                       | 预期耗时 |
| ------------------ | ------------------------------------------ | -------------------------------------------------------------- | :------: |
| **综合诊断** | 输入"康美药业有造假风险吗？"               | 主体确认→编排Agent调度三模块并行→交叉验证→流式输出结论+图表 |   ≤8s   |
| **图谱联动** | 点击报告中"大股东减持"条目                 | 图谱自动聚焦对应节点，高亮减持相关路径                 |   ≤1s   |
| **股权穿透** | 输入"康美实际控制人是谁？"                 | 模糊匹配→候选确认→Neo4j多跳寻路→输出链路图+持股比例         |   ≤3s   |
| **事件溯源** | 输入"康美最近出了什么事？"                 | 公告聚合→情绪统计→事件时间线→因果链标注                     |   ≤3s   |
| **多轮追问** | 在上一轮回答后追问"那它的应收账龄结构呢？" | 记忆Agent提取上文"康美药业"→消解"它"→调财务模块查明细        |   ≤5s   |
| **行业对标** | 追问"它在行业里算什么水平？"               | 取申万行业→计算各指标分位→返回排名+同类公司列表              |   ≤2s   |
| **跨公司对比** | 从对话链接或手动添加多家公司 | 多公司并排对比→指标表格+行业分位可视化+关键差异标注 | ≤3s |
| **报告导出** | 在企业画像页点击"导出报告" | 后台生成→状态轮询→完成后下载PDF，含数据快照+图表+证据 | ≤10s |

> 耗时含LLM推理。单模块查询（股权/舆情/财务/行业）可并行执行，综合诊断耗时 = max(各模块耗时) + 问答Agent汇总耗时。

### 3.2 运行时协作：多维度交叉验证

```
用户提问 → 记忆Agent补上下文（先跑）
         → 主体识别（模糊名称→候选列表→确认目标）
         → 意图匹配的模块先执行
           ├─ 任一模块发现异常 → 自动触发其余两模块链式核查
           │   财务异常 → 查股权(+减持/质押) + 查舆情(+负面公告)
           │   舆情异常 → 查财务(+同期数据是否印证) + 查股权(+控制人变动)
           │   股权异常 → 查财务(+现金流/应收) + 查舆情(+相关事件)
           └─ 无异常 → 各模块按需独立
         → 问答Agent汇总 → 结构化JSON → 前端渲染
```

**一次完整问答时序**：见 [§7.1 一次完整问答时序](#71-一次完整问答时序)。

### 3.3 设计原则

- **计算与解读分离**：Python规则引擎算数字，LLM只做翻译
- **Skill = 说明书+脚本**：每个Skill一个Markdown描述 + 一个pipeline主函数
- **证据链强制追溯**：每条结论必须挂来源（表名+字段+时间段）
- **离线预构建 + 运行时只读**：图谱/行业分位/预计算结果离线跑完，运行时只查询

### 4.10 状态管理

| 状态域             | 关键字段                                                | 说明           |
| ------------------ | ------------------------------------------------------- | -------------- |
| **会话列表** | `sessions: { [id]: { name, messages[], panelData } }` | 多会话独立状态 |
| **当前会话** | `activeSessionId`                                     | 切换时本地恢复 |
| **连接状态** | `wsState`                                             | WebSocket 状态 |

`panelData` 结构——只存四样核心结论：

```
panelData = {
  risk_level: "red",
  rules: [{ rule_id: "R1", triggered: true, severity: "red", ... }],
  timeline: [{ date, title, risk_level, ... }],
  evidence: { financial: [...], related_party: [...], sentiment: [...] }
}
```

数据流：

```
WebSocket.onmessage → dispatch({ type, payload, session_id })
  ├─ thinking    → 实时展示执行进度
  ├─ text_chunk  → 追加到对话区
  ├─ structured  → 写入 panelData，面板即时更新
  ├─ candidates  → 弹出候选公司浮层
  └─ done        → 更新风险等级 + 追问建议
```

### 4.12 风险视觉规范

全平台统一使用五等级颜色编码，确保用户一眼识别风险严重程度：

| 等级 | 颜色 | 色值 | 使用场景 |
|:--:|:--:|------|------|
| 🔴 红色 | 高危 | `#DC2626` | ≥3 条规则触发、监管介入事件、实控人判刑 |
| 🟠 橙色 | 警示 | `#EA580C` | 1-2 条规则触发、评级下调≥2家、股权冻结 |
| 🟡 黄色 | 关注 | `#CA8A04` | 单个异常信号、负面公告密度上升 |
| 🔵 蓝色 | 观察 | `#2563EB` | 数据覆盖不足但无异常、行业分位偏高 |
| 🟢 绿色 | 正常 | `#16A34A` | 无规则触发、无负面事件 |

使用规则：
- 图谱节点颜色 = 该节点的最高风险标签等级；
- 规则卡片颜色 = 该规则的 severity 等级；
- 事件时间线节点颜色 = 事件的 risk_level；
- 始终配文字标签，不只依赖颜色传达信息。

### 4.13 WebSocket 事件处理（Reducer）

前端通过 WS Reducer 统一处理 10 种服务端事件，状态机如下：

| 事件类型 | 触发时机 | Reducer 行为 |
|------|------|------|
| `thinking` | 模块开始执行 | 在面板顶部展示执行进度条 |
| `candidates` | 模糊匹配返回 | 弹出候选公司浮层，等待用户确认 |
| `text_chunk` | 结论流式输出 | 追加文本到对话区当前消息末尾 |
| `rule_card` | 财务规则计算完成 | 写入 panelData.rules，面板即时更新 |
| `equity_graph` | 股权穿透完成 | 写入 panelData.equity，渲染力导向图 |
| `timeline` | 舆情聚合完成 | 写入 panelData.timeline，渲染时间线 |
| `evidence` | 证据链生成 | 写入 panelData.evidence，渲染折叠卡片 |
| `risk_level` | 综合评分完成 | 写入 panelData.risk_level，更新顶部风险条 |
| `done` | 本轮问答结束 | 更新追问建议，关闭进度条 |
| `error` | 模块异常 | 对应区域展示"数据暂缺"，不阻塞其他模块 |

## 5. 后端设计

### 5.1 后端服务设计

#### 5.1.1 编排 Agent

| 属性 | 内容                                                               |
| ---- | ------------------------------------------------------------------ |
| 职责 | 意图识别→主体消歧→工具路由→结果校验→多维度交叉验证触发→自纠错 |
| 框架 | LangGraph StateGraph                                               |
| 输入 | 用户自然语言问题 + 对话历史                                        |
| 输出 | 调度指令（调哪些模块、按什么顺序）                                 |

**交叉验证调度逻辑**：

```
IF 财务勾稽返回异常信号≥1条:
    → 自动触发 股权穿透Skill（查大股东持股变化）
    → 自动触发 舆情事件Skill（查同期负面公告密度）
IF 舆情事件返回负面密度>阈值:
    → 自动触发 财务勾稽Agent（查同期财务是否印证）
    → 自动触发 股权穿透Skill（查控制链变动）
IF 股权穿透返回大股东减持/高质押:
    → 自动触发 财务勾稽Agent（查现金流/应收）
    → 自动触发 舆情事件Skill（查相关报道）
```

#### 5.1.2 对话记忆 Agent

| 属性 | 内容 |
|------|------|
| 职责 | 多轮对话上下文提取、指代消解（"上次那家公司""它"） |

**记忆分为两级**：

| 级别 | 存储 | 保留内容 | 用途 |
|:--:|------|------|------|
| 短期记忆 | LangGraph checkpointer | 完整对话历史 + State 快照 | 当前会话上下文、指代消解 |
| 长期记忆 | ChromaDB（Phase 3 启用） | 关键事实（公司名、指标、时间、结论） | 跨会话记忆持久化、历史分析复用 |

MVP 阶段使用短期记忆——保留最近 N 轮完整上下文，由 LLM 从历史中提取关联实体（公司名/指标/时间）。后续引入 ChromaDB 做长期记忆持久化，支持跨会话复用。

#### 5.1.3 财务勾稽 Agent

| 属性   | 内容                                                |
| ------ | --------------------------------------------------- |
| 职责   | 7条跨科目勾稽规则计算 + LLM解读 + 多维风险评分      |
| 计算层 | Python硬编码公式，100%可复现                        |
| 输出   | 异常信号清单（触警规则+数据对比+行业分位+历史趋势） |


#### 5.1.4 问答生成 Agent

| 属性     | 内容                                                                  |
| -------- | --------------------------------------------------------------------- |
| 职责     | 汇总全模块输出→结构化JSON→前端渲染                                  |
| 输出结构 | 文本结论 + 规则触发清单 + 股权链路 + 事件时间线 + 证据引用 + 追问建议 |

**降级输出规则**：某个模块未执行或执行失败时，不阻塞整体输出：

| 场景               | 处理方式                                                      |
| ------------------ | ------------------------------------------------------------- |
| 模块返回"数据不足" | 对应区域显示"该维度数据暂缺"，其他维度正常展示                |
| 模块执行超时       | 5秒超时→降级返回已有结果，缺失区域标注"分析超时，可稍后重试" |
| 全部模块无异常信号 | 结论输出"未发现明显财务异常信号"，仍展示基本信息+股权结构     |
| 用户问的是简单查询 | 不触发交叉验证，直接返回查询结果即可                          |

> 输出模板见 §5.2

#### 5.1.5 LangGraph State（Agent间共享上下文）

所有Agent读写同一个 State 对象，LangGraph 按节点流转时自动传递。以下为 State 字段定义：

```python
from typing import TypedDict, List, Optional, Annotated
from langgraph.graph.message import add_messages

class AgentState(TypedDict):
    # === 用户输入 ===
    user_query: str                      # 当前用户问题
    session_id: str                      # 会话ID
    confirmed_company: Optional[str]     # 用户确认的目标公司 wind_code

    # === 对话历史 ===
    messages: Annotated[list, add_messages]  # 完整消息历史（LangGraph内置reducer）

    # === 编排控制 ===
    intent: Optional[str]                # 意图分类：diagnose / equity_check / sentiment_check / simple_query
    active_modules: List[str]            # 本轮需要执行的模块列表
    cross_verify_triggered: bool         # 是否已触发交叉验证（防重复触发）
    cross_verify_source: Optional[str]   # 交叉验证由哪个模块触发

    # === 记忆Agent输出 ===
    context_entities: List[dict]         # 从历史中提取的关联实体，如 [{"type":"company","name":"康美药业","code":"600518.SH"}]
    resolved_references: dict            # 指代消解结果，如 {"它":"康美药业"}

    # === 财务勾稽Agent输出 ===
    finance_result: Optional[dict]       # 含触警规则清单+行业分位+历史趋势，格式见 §5.2.4
    finance_error: Optional[str]         # 模块异常信息

    # === 股权穿透Skill输出 ===
    equity_result: Optional[dict]        # 含链路+节点风险标签+关联方异常，格式见 §5.2.5
    equity_error: Optional[str]

    # === 舆情事件Skill输出 ===
    sentiment_result: Optional[dict]     # 含情绪统计+时间线+评级拐点，格式见 §5.2.6
    sentiment_error: Optional[str]

    # === 问答Agent汇总输出 ===
    final_response: Optional[dict]       # 格式见 §5.2，前端渲染用
    risk_level: Optional[str]            # 综合风险等级：红/橙/黄/蓝/绿
```

**State 流转说明**：

- `messages` 使用 LangGraph 内置 `add_messages` reducer，自动追加而非覆盖
- 各模块的 `*_result` 和 `*_error` 互斥——成功写 result、失败写 error
- `cross_verify_source` 记录首次发现异常的模块，用于日志追溯
- State 中的 `Optional` 字段初始值为 `None`，问答Agent汇总时检查非空才渲染

**幂等、取消与恢复**：

- **幂等**：同一 session_id + 同一消息 ID 的重复请求，返回缓存结果而非重新执行；
- **取消**：前端断开 WS 连接时，LangGraph 通过 checkpointer 中断当前执行，不产生脏数据；
- **恢复**：会话恢复时从 checkpointer 最近的 checkpoint 重建 State。

#### 5.2.9 置信度模型

系统对所有关键判断标注置信度，而非只给一个分数：

| 置信度 | 含义 | 示例 |
|:--:|------|------|
| `high` | 多源数据一致，规则明确触发 | 三条规则同时触发+行业分位>90+舆情印证 |
| `medium` | 数据部分覆盖，单一信号 | 仅一条规则触发，行业数据不足 |
| `low` | 数据缺失或信号模糊 | 关键字段为空，或阈值边界附近 |
| `unknown` | 数据完全不可用 | 公司不在数据库覆盖范围内 |

每个 Claim（断言）附带置信度，综合风险等级由所有 Claim 的置信度加权决定。

### 5.3 证据与血缘模型

这是"结构化问答"创新点的技术落地——每项结论都可追溯到具体数据来源。

**EvidenceRef（证据引用）**：

每条证据包含：证据ID、来源类型（资产负债表/公告/研报/股东数据）、来源记录ID、字段路径、报告期、数值+单位、报表口径（母公司/合并）、来源标题、数据集版本、检索时间。

**Claim（断言/结论）**：

每条结论包含：结论文本、结论类型（事实/算法判断/LLM解释/建议）、严重程度、置信度、关联规则ID、引用证据ID列表、验证状态、局限性说明。

**核心规则**：

- 事实型 Claim 至少附带一条 Evidence；
- LLM 只能引用已提供的证据，不能编造数据来源；
- 无证据内容只能作为解释、假设或追问建议；
- Claim 与 Evidence 均带版本和时间戳，支持事后复现。

| 429 | 请求频率超限 | 短时间大量请求 |
| 500 | 服务端异常 | 未预期的内部错误 |

### 6.3 WebSocket 通信协议

#### 6.3.1 Client → Server

| type | 说明 | payload 字段 |
|------|------|------|
| `query` | 用户提问 | `text`(必填): 用户输入文本; `session_id`(必填): 会话ID |
| `confirm_company` | 确认主体 | `wind_code`(必填): 选定的股票代码; `session_id`(必填) |
| `follow_up` | 追问 | `text`(必填): 追问文本; `session_id`(必填) |
| `ping` | 心跳保活 | 无参数 |

**示例**：

```json
{"type": "query", "payload": {"text": "康美药业有财务造假风险吗？", "session_id": "s35_q1"}}
{"type": "confirm_company", "payload": {"wind_code": "600518.SH", "session_id": "s35_q1"}}
{"type": "follow_up", "payload": {"text": "那它的应收账龄结构呢？", "session_id": "s35_q1"}}
{"type": "ping", "payload": {}}
```

#### 6.3.2 Server → Client

| type | 说明 | payload 字段 |
|------|------|------|
| `thinking` | 执行进度 | `step`(string): 当前步骤描述; `module`(string): 当前模块名(equity/finance/sentiment) |
| `text_chunk` | 结论逐字流式 | `text`(string): 文本片段; `is_complete`(bool): 是否完成 |
| `structured_data` | 图表/清单 | `component`(string): 组件名(rule_card/equity_graph/timeline/evidence); `data`(object): 组件数据 |
| `candidates` | 候选公司列表 | `candidates`(array): 候选公司对象数组; `prompt`(string): 提示文字 |
| `done` | 本轮完成 | `risk_level`(string): 综合风险等级; `follow_ups`(array): 追问建议列表 |
| `error` | 异常 | `message`(string): 错误描述; `recoverable`(bool): 是否可恢复 |

**示例**：

```json
{"type": "thinking", "payload": {"step": "正在分析财务报表...", "module": "finance"}}
{"type": "text_chunk", "payload": {"text": "康美药业存在以下需要关注的信号...", "is_complete": false}}
{"type": "structured_data", "payload": {"component": "rule_card", "data": {"rules": [...]}}}
{"type": "structured_data", "payload": {"component": "equity_graph", "data": {"nodes": [...], "edges": [...]}}}
{"type": "structured_data", "payload": {"component": "timeline", "data": {"events": [...]}}}
{"type": "structured_data", "payload": {"component": "evidence", "data": {"financial": [...], "related_party": [...]}}}
{"type": "candidates", "payload": {"candidates": [{"wind_code": "600518.SH", "sec_name": "康美药业", "industry_l1": "中药"}], "prompt": "请选择您要查询的公司"}}
{"type": "done", "payload": {"risk_level": "red", "follow_ups": ["查看康美药业完整画像", "对比同行业公司"]}}
{"type": "error", "payload": {"message": "该公司暂无财务数据", "recoverable": true}}
```

#### 6.3.3 一次完整问答的消息时序

```
Client → { type: "query", payload: { text: "康美有造假风险吗？" } }
Server → { type: "candidates", payload: { candidates: [...] } }       // 模糊匹配时
Client → { type: "confirm_company", payload: { wind_code: "600518.SH" } }
Server → { type: "thinking", payload: { step: "正在分析财务报表...", module: "finance" } }
Server → { type: "thinking", payload: { step: "正在穿透股权关系...", module: "equity" } }
Server → { type: "thinking", payload: { step: "正在汇总舆情事件...", module: "sentiment" } }
Server → { type: "text_chunk", payload: { text: "康美药业...", is_complete: false } }
Server → { type: "structured_data", payload: { component: "rule_card", data: {...} } }
Server → { type: "structured_data", payload: { component: "equity_graph", data: {...} } }
Server → { type: "structured_data", payload: { component: "timeline", data: {...} } }

```

### 6.4 错误码

| 状态码 | 说明                             | 前端处理                                  |
| :----: | -------------------------------- | ----------------------------------------- |
|  200  | 成功                             | 正常渲染                                  |
|  400  | 参数错误（如股票代码格式不正确） | Toast提示，引导用户重新输入               |
|  404  | 该公司不在数据覆盖范围内         | 提示"暂未收录该公司数据"，建议换一家      |
|  429  | 请求频率超限                     | 前端节流，展示冷却倒计时                  |
|  500  | 服务端异常                       | 提示"系统繁忙，请稍后重试"，可重试        |
|  1001  | LLM调用超时                      | 自动重试1次，仍失败则降级返回已有结果     |
|  1002  | Neo4j查询超时                    | 降级——跳过股权穿透，仅展示财务+舆情结果 |
|  1003  | 模块数据不足                     | 标注"该模块数据暂缺"，不影响其他模块返回  |

## 8. 技术选型

为便于多项目统一，选型要求：

- 前端网页：React + Vite
- 后端服务：FastAPI、Python
- 后端数据：MySQL + Neo4j
- 后端模型：DeepSeek V4

### 8.1 前端技术选型

| 技术      | 用途       | 选型理由                   |
| --------- | ---------- | -------------------------- |
| React 18  | UI 框架    | 组件化开发，生态成熟       |
| Vite      | 构建工具   | 启动快，热更新，配置简洁   |
| shadcn/ui | 组件库     | 可定制，轻量，无运行时依赖 |
| Recharts  | 图表库     | React 原生，声明式 API     |
| d3/visx   | 图谱可视化 | 力导向图，可拖拽缩放       |

### 8.2 后端技术选型

| 技术             | 用途       | 选型理由                               |
| ---------------- | ---------- | -------------------------------------- |
| FastAPI          | Web 框架   | 异步高性能，自动 API 文档，Python 生态 |
| LangGraph        | Agent 编排 | 状态图匹配条件分支，支持多节点并行     |
| DeepSeek V4/V3.2 | LLM 推理   | 主模型，成本极低，中文理解强           |
| LLM Provider Adapter | 模型抽象 | 统一接口，支持 DeepSeek/Qwen/Mock 切换，不绑定单一模型 |
| Pydantic V2      | 数据校验   | 类型安全，与 FastAPI 深度集成          |

### 8.3 数据库技术选型

| 技术              | 用途         | 选型理由                                         |
| ----------------- | ------------ | ------------------------------------------------ |
| MySQL 8.4         | 结构化数据   | SQL 标准，三表+公告+研报+股东数据存储            |
| Neo4j 2025.06.1   | 图存储与查询 | 原生图算法（寻路/社群检测/中心度），赛题推荐方案 |
| ChromaDB 0.5.23   | 向量存储     | Python 原生，持久化内置，Phase 3 启用            |
| BGE-small-zh-v1.5 | 嵌入模型     | 中文语义检索，轻量本地部署                       |
